# Notifier Service

Сервис для отправки сообщений с ограничением по частоте запросов.

## Что это делает

Берёт сообщения и отправляет их во внешний API. Не более N запросов в секунду. Если API вернёт 429, повторяет отправку.

## Как работает

### Worker Pool
Запускается несколько горутин (воркеров), каждая слушает канал с сообщениями. Когда новое сообщение приходит, один из свободных воркеров его берёт и обрабатывает.

### Rate Limiting
Используется `time.Ticker` с интервалом `1 секунда / N запросов`. Перед отправкой воркер ждёт тика от лимитера, потом отправляет. Все воркеры читают из одного общего тикера, поэтому лимит глобальный (N запросов в секунду).

Про `time.Ticker`:
- В коде реализована валидация параметров (см. ниже), чтобы избежать деления на ноль при создании тикера.
- `time.Ticker` прост и подходит для задания глобального RPS, но имеет ограничения по точности (drift, системная задержка) и по поведению при очень больших значениях rate. Для более тонкого контроля (burst, Wait(ctx) и т.д.) можно использовать `golang.org/x/time/rate`.

### Retry
Если пришла ошибка или статус 429:
- При ошибке клиента (ошибочный `err`) считаем как failed
- При 429 повторяем (максимум 3 попытки)
- Между попытками ждём: 100ms, потом 200ms, потом 400ms (экспоненциальный backoff)

### Shutdown
При вызове Close():
- Устанавливаем флаг closed 
- Отменяем общий context (останавливаем воркеры)
- Ждём завершения всех воркеров (wg.Wait())
- Останавливаем rate limiter
- Канал jobs НЕ закрывается

Это сделано намеренно, чтобы:
1. избежать race condition (send vs close)
2. избежать panic при отправке в закрытый канал

Поведение после Close():
- Новые вызовы Send() возвращают ошибку "notifier is closed"
- Уже отправленные сообщения могут быть частично обработаны (graceful shutdown)
- Новые сообщения не принимаются

## Структура кода
- notifier.go - основная структура Notifier и методы Send/Close 
- worker.go - логика работы воркеров и обработки сообщений 
- stats.go - счётчики (Sent, Failed, Retries) с атомарными операциями

## Использование

```go
// Создаём notifier с 5 воркерами и 10 запросов в сек
notifier := NewNotifier(client, 5, 10)
defer notifier.Close()

// Отправляем сообщение
ctx := context.Background()
err := notifier.Send(ctx, Message{
    ID:      "123",
    Payload: "hello",
})

// Смотрим статистику
stats := notifier.GetStats()
fmt.Println(stats.Sent, stats.Failed, stats.Retries)
```

NewNotifier делает базовую валидацию входных параметров:
- workers — минимальное значение 1 (если передали <=0, используется 1).

## Тесты
go test -race ./...

Проверяем:

- простая отправка
- отправка после закрытия сервиса
- retry при 429
- если все попытки не прошли
- ошибки клиента
- параллельные отправки
- rate limiting работает
- graceful shutdown

## Особенности
- Потокобезопасность через atomic
- Нет закрытия каналов - нет panic/race
- Graceful shutdown через context
- Поддержка отмены через context в Send
- Экспоненциальный backoff

## Ограничения

Буфер канала jobs = 1000: 
- при перегрузке отправители могут блокироваться

После Close() возможен микроскопический race:
- Send() может успеть отправить сообщение до того, как увидит closed
