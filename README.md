# Notifier Service

Сервис для отправки сообщений с ограничением по частоте запросов.

## Что это делает

Берёт сообщения и отправляет их во внешний API. Не более N запросов в секунду. Если API вернёт 429, повторяет отправку.

## Как работает

### Worker Pool
Запускается несколько горутин (воркеров), каждая слушает канал с сообщениями. Когда новое сообщение приходит, один из свободных воркеров его берёт и обрабатывает.

### Rate Limiting
Используется `time.Ticker` с интервалом `1 секунда / N запросов`. Перед отправкой воркер ждёт тика от лимитера, потом отправляет.

### Retry
Если пришла ошибка или статус 429:
- При ошибке сразу считаем как failed
- При 429 повторяем (максимум 3 попытки)
- Между попытками ждём: 100ms, потом 200ms, потом 400ms (экспоненциальный backoff)

### Shutdown
Когда вызываем `Close()`:
1. Отмечаем что закрыли (`closed.Store(true)`)
2. Отменяем контекст для всех воркеров
3. Закрываем канал с сообщениями
4. Ждём пока все воркеры закончат текущие сообщения (`wg.Wait()`)

## Структура кода
- notifier.go - основная структура Notifier и методы Send/Close 
- worker.go - логика работы воркеров и обработки сообщений 
- stats.go - счётчики (Sent, Failed, Retries) с атомарными операциями

## Использование

```go
// Создаём notifier с 5 воркерами и 10 запросов в сек
notifier := NewNotifier(client, 5, 10)
defer notifier.Close()

// Отправляем сообщение
ctx := context.Background()
err := notifier.Send(ctx, Message{
    ID:      "123",
    Payload: "hello",
})

// Смотрим статистику
stats := notifier.GetStats()
fmt.Println(stats.Sent, stats.Failed, stats.Retries)
```

Проверяем:

- отправка после закрытия сервиса
- retry при 429
- если все попытки не прошли
- ошибки клиента
- параллельные отправки
- rate limiting работает
- graceful shutdown

## Особенности
- Потокобезопасно: используем sync/atomic для счётчиков
- Нет блокировок: только каналы и мьютекс для Send
- Graceful: при закрытии ждём завершения всех текущих задач
- Экспоненциальный backoff: между retry'ями ждём всё дольше

## Ограничения

- Размер буфера канала 1000 - если быстро отправить много, очередь переполнится
- Контекст создаётся как Background - можно передать свой с timeout'ом
- Если воркеры не успевают, сообщения скапливаются в памяти
